tcc 的词法分析(二)

宏只处理 define

做法是如果遇到宏，就跳转过去读，作为最新的 token 然后返回

结构

符号表

  • char *idtable 是一个字符数组，存储着每一个符号。符号之间用 ‘\0’ 分割
  • char *idptr, *idlast 辅助指针
    • idptr 指向字符数组第一个空位置
    • idlast 指向符号表中最后的一个符号(插入新符号的时候，重置该指针)

tinycc 在处理文本的时候，除了运算符和数字，都视为符号

返回的时候 token 表明了符号位于符号表的位置（第几个符号）

  • 从 256 开始记的
  • 前面的属于关键字
  • 因为字符直接使用数值作为类型，因此根据 ascii 从 256 开始是符号(包括关键字)

步骤：

  1. 只要第一个判断属于符号，进入符号的状态机，逐个字符插入符号表
  2. 比较判断是否为关键字，如果是的话，保持 idptr 不变，同时更新 tok
  3. 如果不是关键字的话，更新 idptr
  4. 判断该符号是否位于宏替换的范围，如果是的话，不要立即返回
  5. 记录当前输入流读取位置，到宏栈中（维护了一个栈）
  6. 设置输入流的位置到宏替换的位置
  7. 重新调用 next() 来读取替换的符号
  8. 直到行尾，替换结束，根据宏栈的地址，跳转回来（允许嵌套宏）

宏

步骤：

  1. 如果当前符号是 #, 进入分支
  2. 取下一个符号，如果下一个符号就是 define 关键字（固定 token 类型）
  3. 读取下一个符号到符号表中，同时更新 vat 和 vac, 在该符号位置记录了符号类型和数值
  • 对于 define 的符号类型就是 VT_DEFINE, 数值就是文件流的位置，就是现在
  1. 一直跳过字符不再读取了，这部分是宏替换的内容。直到读取完换行符为止

tcc’s first commit

tcc 的词法分析(一)

一个语言的词法分析，三个状态图就可以了：

  1. 标识符
  2. 常量
  3. 符号

tinycc 的常量甚至不做处理，只进行了标识符和符号的识别

  • 可能是在词法分析的地方再处理

空白

利用一个死循环，持续探测到非空白时才退出

while(1) {
    c = inp();
  • 每次进入的时候，需要取一次字符
} else if (c != ' ' & c != 9)
  break;
  • 退出条件

标识符

关键字区分：利用了常见的做法

  • 提前将关键字放入到符号表中
  • 使用一个状态图来进行判断
  • 如果是标识符，再判断符号表中是否已经存在该符号了

巧妙的地方：

  • 标识符的 TOKEN 类型标识不是乱取的
  • 而是符号表中，该符号对应的位置索引
  • 通过该标识符的 TOKEN 数值，既可以判断类型，也可以获取地址

符号

对于单符号和双符号，利用 ungetc 来纠错

巧妙的地方：

  • 对于大部分的单符号，直接用符号值表示类型（同样可以获取到值）
  • 对于双符号和 <, > 直接用二进制码表示类型，方便代码生成？

加速

对于用到数值的地方，尽量做到一值两用，甚至三用

  1. 减少查表：符号表中，标识符直接对应于符号表的索引
  2. 减少转换：运算符中，类型直接就是运算符
  3. 减少映射：生成代码的时候不用类型重新映射二进制代码了，类型就是二进制代码
